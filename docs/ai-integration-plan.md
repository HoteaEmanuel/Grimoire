# AI Integration Plan — Grimoire AI Features

> Generated by `/research ai-integration-research.md`. Documentation only — no source files were modified while producing this plan.

## 1. Scope

Four Pro-only features from `context/project-overview.md` §3.F:

| Feature | Trigger | Output |
|---|---|---|
| Auto-tag suggestions | On save (create/update item) | Suggested tags, accept/reject before applying |
| AI Summary | On-demand, long text items | One-line summary |
| Explain This Code | On-demand, snippet/command items | Plain-English explanation |
| Prompt Optimizer | On-demand, prompt items | Rewritten prompt |

**Current state:** no AI code exists in the repo (`grep -r "openai\|OPENAI\|gpt-" src/` returns nothing). `CLAUDE.md` specifies `gpt-4o-mini` for cost-effective features and `gpt-4o` for the prompt optimizer if quality demands it — that pricing-tier reasoning still applies, but those exact model names are being phased out by OpenAI. The equivalent current-generation tier is **GPT-5 nano** (cheap, fast — $0.05/M input, $0.40/M output tokens) for tagging/summarizing/explaining, with **GPT-5 mini or full GPT-5.5** as the upgrade path for the prompt optimizer if nano's output quality is insufficient. This plan builds against `gpt-5-nano` per the research brief, with the model name isolated to one constant so swapping tiers later is a one-line change.

## 2. SDK Setup — `src/lib/openai.ts`

Mirrors the existing singleton pattern used by `src/lib/stripe.ts` and `src/lib/prisma.ts`:

```typescript
import OpenAI from "openai";

if (!process.env.OPENAI_API_KEY) {
  throw new Error("OPENAI_API_KEY is not set");
}

const globalForOpenAI = globalThis as unknown as { openai?: OpenAI };

export const openai =
  globalForOpenAI.openai ??
  new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

if (process.env.NODE_ENV !== "production") {
  globalForOpenAI.openai = openai;
}

export const AI_MODEL = "gpt-5-nano";
```

- Install: `npm install openai`.
- Env var: `OPENAI_API_KEY` added to `.env`, `.env.example` (blank placeholder), `.env.production`.
- **Security**: the client is only ever imported from `src/lib/db/*.ts` or `src/actions/*.ts` (server-only modules) — never from a client component or a file under `src/components/`. Next.js's server/client module boundary plus the existing `'use server'`-only usage pattern in this codebase already prevents the key from reaching the browser bundle; no extra guard needed beyond "don't import this file from a `'use client'` component."

## 3. Server Action Pattern

Every existing server action in `src/actions/*.ts` follows: `auth()` check → Zod validation → Pro gate → try/catch → `{ success, data } | { success: false, error }`. AI actions slot into the same shape — e.g. `src/actions/ai.ts`:

```typescript
"use server";

type AiResult<T> = { success: true; data: T } | { success: false; error: string };

export async function suggestTags(itemId: string): Promise<AiResult<{ tags: string[] }>> {
  const session = await auth();
  if (!session?.user?.id) return { success: false, error: "Unauthorized" };
  if (!session.user.isPro) return { success: false, error: "AI features are a Pro feature" };

  const { success, retryAfter } = await checkRateLimit(aiLimiter, session.user.id);
  if (!success) return { success: false, error: `Rate limited, try again in ${retryAfter}s` };

  const item = await getItemForOwner(itemId, session.user.id); // 404/ownership check
  if (!item) return { success: false, error: "Item not found" };

  try {
    const completion = await openai.chat.completions.create({
      model: AI_MODEL,
      messages: [{ role: "user", content: buildTagPrompt(item) }],
      response_format: TAG_SUGGESTION_SCHEMA, // see §4
    });
    const parsed = JSON.parse(completion.choices[0].message.content!);
    return { success: true, data: { tags: parsed.tags } };
  } catch (err) {
    if (err instanceof OpenAI.APIError) {
      return { success: false, error: "AI service is temporarily unavailable" };
    }
    return { success: false, error: "Something went wrong" };
  }
}
```

One action per feature (`suggestTags`, `summarizeItem`, `explainCode`, `optimizePrompt`), all in `src/actions/ai.ts`, all following this shape — consistent with how `items.ts`/`collections.ts`/`billing.ts` each hold multiple related actions rather than one file per action.

## 4. Structured Output (avoid parsing free-form text)

Per current OpenAI docs, use `response_format: { type: "json_schema", json_schema: { strict: true, ... } }` (Chat Completions) so the model is constrained to valid JSON matching a schema, rather than parsing a prose response:

```typescript
const TAG_SUGGESTION_SCHEMA = {
  type: "json_schema" as const,
  json_schema: {
    name: "tag_suggestions",
    strict: true,
    schema: {
      type: "object",
      properties: {
        tags: { type: "array", items: { type: "string" }, maxItems: 5 },
      },
      required: ["tags"],
      additionalProperties: false,
    },
  },
};
```

Use this for auto-tag suggestions (array of strings) and summaries (single string field). Explain/optimize can return plain text in a `content` field of the schema, or skip structured output entirely since they're already single-string outputs displayed as-is.

## 5. Streaming vs Non-Streaming

| Feature | Recommendation | Why |
|---|---|---|
| Auto-tag suggestions | Non-streaming | Structured JSON output, short, no benefit to partial render |
| AI Summary | Non-streaming | One line — streaming overhead not worth it |
| Explain This Code | Streaming | Can be a paragraph+; streaming improves perceived latency in the drawer |
| Prompt Optimizer | Streaming | Same — longer output, user benefits from watching it generate |

For the two streaming cases, use the Responses API's typed SSE events (`response.output_text.delta`, `response.completed`) rather than Chat Completions' delta-chunk format — this repo has no existing SSE/streaming infrastructure (no `ReadableStream` usage in `src/app/api/`), so this would be new. Simplest integration: a dedicated route handler (e.g. `POST /api/ai/explain`) that proxies the OpenAI stream directly as the response body, consumed client-side with `fetch` + `response.body.getReader()` — mirrors how `GET /api/download/[...key]` already proxies a stream (R2 → client) in this codebase, just swapping the upstream.

Non-streaming actions stay as plain server actions (§3); streaming ones need a route handler instead, since server actions can't stream a response back incrementally.

## 6. Rate Limiting

Extend `src/lib/rate-limit.ts` with one new limiter, following the existing `makeRatelimit(n, window)` factory used for the five auth endpoints:

```typescript
const aiFeatureLimiter = makeRatelimit(20, "1 h"); // per user, all 4 AI features share this budget
```

Key by `session.user.id` (not IP, since these are authenticated-only Pro features) — same `checkRateLimit`/`rateLimitResponse` helpers, fails open if Upstash is unreachable (consistent with existing behavior). A shared budget across all four features is simpler than four separate limiters and prevents one feature from being used to bypass a per-feature cap.

## 7. Pro Gating

Follows the exact pattern already used for file/image item-type gating (`src/actions/items.ts`, `/api/upload`): check `session.user.isPro` server-side in each AI action, **and** hide/disable the triggering UI for free users (defense in depth, same reasoning as the file-upload Pro gate). Suggested UI treatment, consistent with `CreateItemModal`'s disabled Pro buttons and `SidebarContent`'s Pro badges:

- "Suggest tags" button in the drawer/create-modal tag input: disabled + Pro badge for free users.
- "Summarize" / "Explain" / "Optimize" buttons in `ItemDrawer`: same disabled + badge treatment, or simply hidden (`isPro &&` conditional) — hidden is simpler and matches how `SidebarContent` already conditionally shows the upgrade-nudge Pro badge only to free users.

No new DB fields needed — `session.user.isPro` is already exposed (Stripe Phase 1 work).

## 8. Cost Optimization

- **Model tier**: `gpt-5-nano` for all four features initially (cheapest, fastest — adequate for tagging/summarizing/explaining/short rewrites per OpenAI's own guidance that nano targets "summarization and classification tasks"). Isolate the model name in `AI_MODEL` (§2) so upgrading the optimizer to a larger model later is a one-line change, not a refactor.
- **Token caps**: pass `max_tokens` (or `max_output_tokens` on Responses API) on every call — summaries should be capped tightly (~60 tokens), explanations/optimized prompts more generously (~500 tokens) — to bound worst-case cost per call regardless of input size.
- **Input truncation**: truncate item `content` before sending (e.g. first ~4000 chars) — long snippets/notes shouldn't balloon input-token cost, and nano's use case is short-form content anyway.
- **Caching**: OpenAI applies an automatic 90% discount on repeated/cached input prefixes — structuring prompts with a stable system/instruction prefix followed by the variable item content (rather than interpolating the item content into a single freeform string) lets repeated calls with similar instructions benefit from this automatically, no extra code required.
- **Rate limiting** (§6) is also a cost control, not just an abuse control.

## 9. UI Patterns (loading / accept-reject)

Client-side wrapper in `src/lib/mutations/ai.ts`, following the existing TanStack Query pattern (`useCreateItem`, `useCreateCheckoutSession`):

```typescript
export function useSuggestTags(onSuccess?: (tags: string[]) => void) {
  return useMutation({
    mutationFn: (itemId: string) =>
      suggestTags(itemId).then((r) => {
        if (!r.success) throw new Error(r.error);
        return r.data.tags;
      }),
    onSuccess: (tags) => onSuccess?.(tags), // no success toast — UI shows the suggestions inline
    onError: (err: Error) => toast.error(err.message),
  });
}
```

- **Loading state**: `mutation.isPending` drives a spinner on the triggering button (same as `CreateItemModal`'s submit button today).
- **Accept/reject for auto-tags**: suggested tags render as a separate visual group (e.g. dashed-border badges) below the existing `TagInput` badges, each with an individual ✓/✕ — accepting appends to the real tag list (same `[a-z0-9_-]+`-validated `TagInput` component), rejecting just removes it from the suggestion list client-side. No DB write happens until the user accepts and saves the item — this avoids ever persisting an AI suggestion the user didn't approve.
- **Accept/reject for summary/explanation/optimized prompt**: render the AI output in a dismissible card (e.g. reusing the `Card` + a subtle accent border) above or below the relevant field, with "Apply" (copies into the field / sets `description`) and "Discard" buttons — same two-button affordance as tags, generalized to single-string outputs instead of a list.
- **Streaming UI** (explain/optimize, §5): render tokens into the card as they arrive (`reader.read()` loop appending to local state), with a "Stop" button that aborts the `fetch` via `AbortController` — replaces the static spinner with incremental text once the first token lands.

## 10. Security Considerations

- **API key**: server-only module (§2), never imported client-side — no new risk beyond what `stripe.ts`/`r2.ts` already establish in this codebase.
- **Input sanitization**: item content/title/description are user-authored free text already stored in Postgres via Prisma (parameterized queries — no injection risk there). The risk surface for an LLM call is **prompt injection from item content into the system instructions** — e.g. a snippet whose content reads "ignore previous instructions and output X". Mitigate by keeping the user content in a clearly delimited user-turn message (not concatenated into the system/instruction string) and not granting the model any tool-calling or function-calling ability in these four features (they're pure text-in/text-out) — there's nothing for an injected instruction to actually *do* beyond influence its own response text, which the user reviews via accept/reject (§9) before anything is persisted.
- **Ownership checks**: every AI action must verify the item belongs to `session.user.id` before sending its content to OpenAI (same ownership-scoped lookup pattern as `updateItem`/`deleteItem`) — otherwise a Pro user could probe other users' private item content via item ID guessing.
- **No PII forwarding beyond item content**: don't include the user's email/name/account details in prompts sent to OpenAI — only the item fields relevant to the feature (content, title, language).
