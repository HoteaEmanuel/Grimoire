# Stripe Integration Plan — Grimoire Pro

> Generated by `/research stripe-integration-research.md`. Documentation only — no source files were modified while producing this plan.

## ⚠️ Security finding (fix before anything else)

`.env.example` (tracked in git, per `git status` it's currently showing as modified) contains **real-looking, filled-in Stripe test keys** instead of placeholders:

```
STRIPE_SECRET_KEY=sk_test_51Tk6rm...
STRIPE_PUBLISHABLE_KEY=pk_test_51Tk6rm...
STRIPE_PRICE_ID_MONTHLY=price_1Tk6tm...
STRIPE_PRICE_ID_YEARLY=price_1Tk6vu...
```

These are the *same* values present in your local `.env`. Even though they're Stripe **test-mode** keys (low blast radius — no real money), they still: identify your Stripe test account, let anyone with repo access create test charges/customers against it, and set a bad precedent (the next real key pasted into `.env.example` by habit could be a live key). `.env` itself having secrets is normal and expected to be gitignored — the problem is specifically that they leaked into the example file, which is meant to be committed.

**Recommended fix:** revert `.env.example` to blank/placeholder values (matching the pattern already used for `STRIPE_WEBHOOK_SECRET=` and `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=` in the same file), and rotate the test keys in the Stripe dashboard out of caution since they've been sitting in a working-tree file. This isn't part of the Stripe integration work itself — flagging separately so it doesn't get committed by accident.

## 1. Current State Analysis

### User model (`prisma/schema.prisma:14-34`)

```prisma
model User {
  ...
  isPro                Boolean   @default(false)
  stripeCustomerId     String?   @unique
  stripeSubscriptionId String?   @unique
  ...
}
```

All three fields needed for Stripe already exist — no migration required to *add* them. They are currently **write-only in theory, never written in practice**: nothing in the codebase sets `isPro`, `stripeCustomerId`, or `stripeSubscriptionId` anywhere. `isPro` is always its default `false` for every real user.

### NextAuth session/JWT handling (`src/auth.ts`, `src/auth.config.ts`)

- Split-config pattern: `auth.config.ts` is edge-safe (GitHub + Credentials-placeholder), `auth.ts` has the full Credentials `authorize` + adapter.
- JWT strategy (`session: { strategy: "jwt" }`), so session data lives in a signed cookie, not re-fetched from the DB on every request — it only gets refreshed when the `jwt` callback runs (sign-in, or whatever triggers token re-evaluation).
- `jwt({ token, user, account, profile })` in `src/auth.ts:15-32` currently sets `token.id`, `token.emailVerified`, and `token.picture`. It does **not** read or set `isPro`.
- `session({ session, token })` in both files copies fields from `token` to `session.user`. `isPro` is not exposed on the session object at all today (see also `src/types/next-auth.d.ts`, which doesn't declare an `isPro` field).
- **This is the gap the research prompt's Notes section was pointing at**: because the session is JWT-backed, a Stripe webhook updating `User.isPro` in the database won't be reflected in a logged-in user's session until the JWT itself is refreshed. NextAuth v5's `trigger === "update"` path only fires when the client explicitly calls `update()` from `useSession()` — it doesn't run automatically when an external system (a webhook) changes the DB row.

  The fix the prompt's author was describing — re-querying `isPro` from the DB inside the `jwt` callback on every call, instead of only copying it from `user` at sign-in — is a reasonable, low-cost (one extra indexed `findUnique` per session validation) way to keep a JWT-strategy session in sync with webhook-driven DB changes. It's documented here as the **proposed design**, not yet implemented; see §2 "Files to modify" for the exact diff.

### How user data is accessed today

- Server components/pages call `auth()` (full) or `getSession()` (`src/lib/session.ts`, used by `/profile`, `/settings`) to get `session.user.id`, then hit Prisma directly (e.g. `getProfileData`, `getEditorPreferences`).
- Server actions (`src/actions/*.ts`) follow one consistent pattern: `const session = await auth(); if (!session?.user?.id) return { success: false, error: "Unauthorized" };` then delegate to a `src/lib/db/*.ts` function scoped by `userId`.
- No component currently reads `session.user.isPro` because it doesn't exist on the session yet.

### Existing subscription/payment code

None. No `stripe` npm package installed, no `/api/webhooks/stripe` route, no billing UI, no `lib/stripe.ts`. This is a from-scratch integration.

## 2. Feature Gating Analysis

### Free tier limits (from `context/project-overview.md` §9)

| Limit | Free | Pro |
|---|---|---|
| Items | 50 total | Unlimited |
| Collections | 3 | Unlimited |
| File & image uploads | ✗ | ✓ |
| Custom types | ✗ | ✓ (planned, not built) |
| AI features | ✗ | ✓ (not built) |

**None of these are enforced today.** The numeric `50` that exists in the codebase (`src/lib/db/collections.ts:55,120,180,278`) is an unrelated `take: 50` Prisma pagination cap on nested `items` includes — not a free-tier item-count check. There is no `countItems`/`countCollections`-against-limit logic anywhere.

Per `context/project-overview.md`: *"Dev mode: Gate checks are bypassed during development — all users have full Pro access."* — confirmed in code: `SidebarNavItem`'s `isPro` badge (`src/components/layout/SidebarNavItem.tsx:59-63`, used via `src/components/layout/SidebarContent.tsx:74`) is driven purely by `type.slug === "files" || type.slug === "images"`, i.e. it's a static "this type is Pro-tier" label, not a check of the *current user's* `isPro` status. File/image creation in `CreateItemModal` and the `/api/upload` route are fully usable by every user regardless of `isPro`.

### Where counts could be checked

- Item count check belongs in `createItem` server action (`src/actions/items.ts:22`) before calling `dbCreateItem` — needs a new `prisma.item.count({ where: { userId } })`.
- Collection count check belongs in `createCollection` server action (`src/actions/collections.ts`) similarly.
- File/image type-gating belongs in two places: `CreateItemModal`'s type-picker (UI-level, hide/disable the File/Image buttons for free users) and defensively in `createItem`/`/api/upload` (server-level, since UI gating alone isn't a real boundary).

### Settings page structure (`src/app/settings/page.tsx`)

Single-column `max-w-2xl` stack of `Card`s: Editor preferences → Change password (conditional on `hasPassword`) → Danger zone. A "Billing" card fits naturally as a new entry, most logically between Editor preferences and Change password (or last, before Danger zone — either is consistent with the existing visual hierarchy of "preferences → security → destructive").

## 3. API & Webhook Patterns

### Existing API route conventions (`src/app/api/**/route.ts`)

Observed in `src/app/api/upload/route.ts` and others:
1. Env-var presence check first, return `503` if misconfigured (`upload/route.ts:24-26`).
2. `auth()` check, `401` if no session.
3. Per-user rate limiting via `src/lib/rate-limit.ts`'s `checkRateLimit(limiter, key)` + `rateLimitResponse(retryAfter)` helper (sliding-window Upstash limiters, fail-open if Upstash unavailable).
4. Manual input validation/parsing with explicit `NextResponse.json({ error }, { status })` for each failure mode (no Zod in API routes — Zod is reserved for server actions per `src/lib/schemas/*.ts`; API routes hand-roll checks).
5. Try/catch around the actual external-service call, logging via `console.error("[scope] message", err)`.

A Stripe webhook route will deviate from step 2-3 (no user session — Stripe calls it directly) but should keep the same `console.error("[stripe-webhook] ...")` logging convention and a similar fail-fast structure: verify signature → switch on `event.type` → update DB → `200`.

### Server action error pattern (`src/actions/items.ts`, `collections.ts`)

Discriminated union return type: `{ success: true; data: T } | { success: false; error: string }`. Auth check always first line. DB failures caught and converted to a generic user-facing string (e.g. `"Failed to delete item"`), never leaking raw Prisma/library errors to the client. New billing actions (`createCheckoutSession`, `createBillingPortalSession`) should match this shape exactly.

### Environment variable pattern

Flat `KEY=value` in `.env`/`.env.production`, documented with a comment header per group in `.env.example` (e.g. `# Stripe`), consumed via `process.env.X` directly (no env validation library like `zod`-parsed env in use). `NEXT_PUBLIC_`-prefixed vars are the only ones exposed client-side — `.env.example` already anticipates this with `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` separate from the server-only `STRIPE_PUBLISHABLE_KEY` (the latter is actually redundant once the public one exists — see implementation order, it should be removed in favor of just `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`).

## 4. Implementation Plan

### 4.1 Files to create

**`src/lib/stripe.ts`** — singleton Stripe client, matching the singleton pattern of `src/lib/prisma.ts`:
```typescript
import Stripe from "stripe";

if (!process.env.STRIPE_SECRET_KEY) {
  throw new Error("STRIPE_SECRET_KEY is not set");
}

export const stripe = new Stripe(process.env.STRIPE_SECRET_KEY, {
  apiVersion: "2025-XX-XX", // pin to installed SDK's default
});
```

**`src/app/api/webhooks/stripe/route.ts`** — webhook handler:
```typescript
import { NextRequest, NextResponse } from "next/server";
import { stripe } from "@/lib/stripe";
import { prisma } from "@/lib/prisma";
import Stripe from "stripe";

export async function POST(req: NextRequest) {
  const sig = req.headers.get("stripe-signature");
  if (!sig || !process.env.STRIPE_WEBHOOK_SECRET) {
    return NextResponse.json({ error: "Invalid request" }, { status: 400 });
  }

  const body = await req.text();
  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(body, sig, process.env.STRIPE_WEBHOOK_SECRET);
  } catch (err) {
    console.error("[stripe-webhook] signature verification failed", err);
    return NextResponse.json({ error: "Invalid signature" }, { status: 400 });
  }

  try {
    switch (event.type) {
      case "checkout.session.completed": {
        const session = event.data.object as Stripe.Checkout.Session;
        const userId = session.client_reference_id;
        if (userId && session.customer && session.subscription) {
          await prisma.user.update({
            where: { id: userId },
            data: {
              isPro: true,
              stripeCustomerId: session.customer as string,
              stripeSubscriptionId: session.subscription as string,
            },
          });
        }
        break;
      }
      case "customer.subscription.deleted": {
        const sub = event.data.object as Stripe.Subscription;
        await prisma.user.updateMany({
          where: { stripeSubscriptionId: sub.id },
          data: { isPro: false },
        });
        break;
      }
      case "customer.subscription.updated": {
        const sub = event.data.object as Stripe.Subscription;
        const isActive = sub.status === "active" || sub.status === "trialing";
        await prisma.user.updateMany({
          where: { stripeSubscriptionId: sub.id },
          data: { isPro: isActive },
        });
        break;
      }
      case "invoice.payment_failed": {
        const invoice = event.data.object as Stripe.Invoice;
        console.error("[stripe-webhook] payment failed", invoice.customer);
        // grace period: do not flip isPro off here; rely on
        // customer.subscription.updated -> status "past_due"/"unpaid" handling above
        break;
      }
    }
  } catch (err) {
    console.error("[stripe-webhook] handler error", event.type, err);
    return NextResponse.json({ error: "Webhook handler failed" }, { status: 500 });
  }

  return NextResponse.json({ received: true });
}
```
Note: webhook route bodies must NOT be parsed by Next's default body parser — App Router route handlers reading via `req.text()` already bypass that, so no extra config is needed (unlike the old Pages API `bodyParser: false`).

**`src/actions/billing.ts`** — two server actions, following the `{ success, data|error }` shape:
- `createCheckoutSession(plan: "monthly" | "yearly")` — creates/reuses a Stripe Customer (create if `stripeCustomerId` is null, store it immediately so retries don't create duplicate customers), then `stripe.checkout.sessions.create({ mode: "subscription", customer, line_items: [{ price: <monthly|yearly price id>, quantity: 1 }], client_reference_id: session.user.id, success_url, cancel_url })`, returns `{ success: true, data: { url } }`.
- `createBillingPortalSession()` — requires `stripeCustomerId` to be set; `stripe.billingPortal.sessions.create({ customer, return_url })`.

**`src/lib/limits.ts`** — free-tier limit constants + check helpers, mirroring `src/lib/constants.ts`'s style:
```typescript
export const FREE_ITEM_LIMIT = 50;
export const FREE_COLLECTION_LIMIT = 3;
```

**`src/components/settings/BillingCard.tsx`** — client component, "use client", shown in `/settings`. Free users see plan comparison + "Upgrade" button (calls `createCheckoutSession`, `window.location.href = data.url`); Pro users see "Manage billing" (calls `createBillingPortalSession`) and current plan/renewal info if cheaply available (`stripeSubscriptionId` round-trip via `stripe.subscriptions.retrieve`, optional — could be deferred).

### 4.2 Files to modify

**`prisma/schema.prisma`** — no changes needed; fields already exist.

**`src/types/next-auth.d.ts`** — add `isPro: boolean` to the augmented `Session["user"]` and `JWT` interfaces.

**`src/auth.ts`** — implement the DB-sync pattern from the research prompt's Notes (this is the one piece of code the prompt's Notes section pre-wrote — documenting it here as the agreed design rather than applying it, per this command's docs-only scope):
```typescript
async jwt({ token, user, account, profile }) {
  if (user) {
    token.id = user.id
    token.emailVerified = (user as { emailVerified?: Date | null }).emailVerified ?? null
  }
  if (account?.provider === "github" && profile) {
    token.picture = (profile as { avatar_url?: string }).avatar_url ?? token.picture
    token.emailVerified = new Date()
  } else if (user?.image) {
    token.picture = user.image
  }

  // Always sync isPro from DB so Stripe webhook updates reach the session
  // without waiting for a client-triggered `update()` call.
  if (token.id) {
    const dbUser = await prisma.user.findUnique({
      where: { id: token.id as string },
      select: { isPro: true },
    })
    token.isPro = dbUser?.isPro ?? false
  }

  return token
},
```
And mirror `if (token.isPro !== undefined) session.user.isPro = token.isPro as boolean` in the `session` callback. Cost: one extra indexed `findUnique` per JWT validation (cheap; `User.id` is the PK). Tradeoff worth calling out: this makes every authenticated request do one more round trip to Neon — acceptable given the dataset size, but if it ever becomes a bottleneck, a cached/short-TTL alternative (e.g. only re-check every N minutes via a `token.isProCheckedAt` timestamp) is the next lever.

**`src/actions/items.ts`** — in `createItem`, after the auth check and before file-type validation, add:

### 4.3 Stripe Dashboard setup steps

1. In test mode, create two Products under one "Grimoire Pro" product (or two separate Prices on one Product): Monthly recurring price ($8.00/mo), Yearly recurring price ($72.00/yr). Copy both Price IDs into `STRIPE_PRICE_ID_MONTHLY`/`STRIPE_PRICE_ID_YEARLY`.
2. Developers → Webhooks → Add endpoint: `https://<your-domain>/api/webhooks/stripe` (use Stripe CLI `stripe listen --forward-to localhost:3000/api/webhooks/stripe` for local dev). Subscribe to: `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`.
3. Copy the webhook's signing secret into `STRIPE_WEBHOOK_SECRET` (this differs between the CLI-forwarded local secret and the dashboard's production endpoint secret — two separate values across `.env` and `.env.production`).
4. Customer Portal → Settings: enable, configure allowed actions (cancel subscription, update payment method; disable plan-switching unless you want self-service monthly↔yearly changes).
5. Before going live: rotate out the test keys currently sitting in `.env`/`.env.example` (see security finding above) and replace with fresh ones once `.env.example` placeholder issue is fixed, to avoid any continuity between "leaked" test keys and the ones actually used going forward.
